# iOS App 專案開發框架（通用版）

> 從 Home Bar（Expo + Supabase）、WedSnap（Expo + Supabase + Next.js）與 DuoRatio（原生 Swift + AVFoundation）三個專案的實戰經驗提煉而成。
>
> **使用方法**：先讀本文件建立文件骨架與 App Store Checklist，再根據技術棧選讀對應附錄：
> - Expo + Supabase → `01-Tech-Expo-Supabase.md`
> - 原生 Swift → `02-Tech-Native-Swift.md`

---

## 一、CLAUDE.md 框架（AI 行為準則）

> 每個新專案都應建立 `CLAUDE.md`，作為 AI 的 System Prompt 級別設定。

### 1.1 AI 角色定義（模板）

```markdown
## AI 角色定義

你是一位資深的 {技術棧描述} 工程師。你的首要任務是寫出穩定、安全、符合 iOS 審核規範的程式碼，並嚴格遵守本文件所述之架構與規範。

### 文件連動機制
- 每次完成一個功能的 Commit 後，你必須主動檢視 TODO.md，並詢問使用者是否需要幫忙勾選已完成項目或更新進度。
- 若在開發過程中發現 PRD.md 或 TECHNICAL_DOCS.md 的設計有遺漏或與實作不符，必須主動提出並協助更新文件。
- 涉及 App Store 審核相關的修改，須同步檢查 PRD.md 的檢查清單。

### Context 載入策略
> 新對話開頭使用：
>
> 「這是本專案的基礎架構文件。請先完整閱讀並記住 CLAUDE.md 的規範。接下來的任務我會指定你讀取 PRD.md 的特定章節與 TODO.md 的進度來進行開發。」
```

**技術棧描述範例**：
- Expo + Supabase：「React Native (Expo) 與 Supabase 全端工程師」
- 原生 Swift：「iOS 原生開發工程師，精通 Swift、SwiftUI、UIKit 與 Apple 框架」

### 1.2 工作流與 Git 規範

```markdown
## 工作流與 Git 規範（Claude 行為準則）

> 以下規則為最高優先級，必須嚴格遵守。

### Git 版本控制與 Commit 規範
- 當使用者輸入「幫我 commit」時，必須先執行 `git status` 與 `git diff`，完整閱讀所有變更後再進行後續動作。
- 嚴格遵守 Conventional Commits：`feat`、`fix`、`refactor`、`style`、`docs`、`chore`、`test`。
- Commit Message 標題使用英文簡述，格式為 `type: short description`。
- 若有較複雜的邏輯改動，內文使用繁體中文條列式說明。
- 撰寫好 Commit Message 後，先向使用者確認，同意後才執行 `git add` 與 `git commit`。

### 語言與溝通規範
- 與使用者對話、程式碼註解、文件一律使用繁體中文。
- 絕對不可出現簡體中文字。

### 專案技術棧原則
- 修改任何程式碼前，必須先閱讀相關檔案，確保理解現有邏輯再動手。
<!-- 技術棧專屬原則請參考對應的技術棧附錄 -->
```

### 1.3 安全性規範（通用）

```markdown
### 安全性規範
- 金鑰與憑證絕對不可寫入程式碼，一律透過環境變數或平台設定管理。
- 錯誤訊息不可洩漏敏感資訊（stack trace、DB 結構、內部 ID）。
- 所有影片/照片等使用者資料須妥善處理，遵守隱私政策承諾。
```

### 1.4 程式碼品質規範（通用）

```markdown
### 程式碼品質規範
- 禁止 debug log 出現在正式程式碼中（console.log / print）。
- 使用統一的 Logger 模組處理錯誤記錄（開發階段輸出至 console，正式版可轉送監控服務）。
- 所有非同步操作必須妥善處理錯誤（try-catch / do-catch）。
```

## 五、App Store 上架完整檢查清單

> 此清單從 Home Bar、WedSnap、DuoRatio 三個專案的實戰經驗整理，適用於所有 iOS App 上架。

### 5.1 Apple 強制要求（缺一退件）

| # | 項目 | 說明 |
|---|------|------|
| 1 | **Sign in with Apple** | 若 App 提供任何第三方登入（Email、Google），必須同時提供 Apple Sign-in，且 UI 優先顯示 |
| 2 | **帳號刪除功能** | 若 App 支援帳號建立，必須提供帳號刪除（不只是登出） |
| 3 | **隱私權政策** | 必須提供公開 URL，填入 App Store Connect 與 App 內。建議獨立網頁部署（非 GitHub Gist） |
| 4 | **服務條款**（若有付費功能）| 含使用者上傳內容的 App 建議提供服務條款頁面 |
| 5 | **隱私「營養標籤」** | 在 App Store Connect 如實填寫資料收集項目（詳見 5.5） |
| 6 | **測試帳號** | 審核備註欄必須提供可登入的測試帳號密碼，**用英文撰寫** |
| 7 | **權限字串** | NSCameraUsageDescription 等，須清楚說明用途 |
| 8 | **權限字串本地化** | 權限彈窗語言須與 App 語言一致（透過 InfoPlist.xcstrings 或 locales/*.json） |

### 5.2 內容與 UI 要求

| # | 項目 | 說明 |
|---|------|------|
| 9 | **內容充足** | 不可有空白頁面、半成品功能、placeholder 文字 |
| 10 | **所有頁面正常運作** | 審核員會點擊每個頁面，須透過 TestFlight 真機測試 |
| 11 | **無權限時的處理** | 拒絕權限時顯示引導提示，非白屏或 crash |
| 12 | **空狀態** | 列表無資料時須有友善提示 |
| 13 | **閃退防護** | 所有非同步操作必須 try-catch |
| 14 | **年齡限制內容** | 酒精/菸草相關 App 須實作年齡驗證閘 + 評級設為 17+ |

### 5.3 付費功能要求

| 模式 | 要求 |
|------|------|
| **一次性付費（Paid App）** | 在 App Store Connect 設定售價即可，App 內不需額外處理 |
| **免費 + 訂閱制** | IAP 產品設定、免費試用條款標示、訂閱管理入口、還原購買按鈕 |
| **免費 + IAP** | IAP 產品設定、還原購買按鈕 |

訂閱制額外要求：
- 付費牆必須清楚標示：「X 天免費試用，之後每月/年 $X.XX」
- App 內必須提供「管理訂閱」連結
- 必須提供「還原購買」按鈕
- 訂閱制 App 應設定為「免費」下載

### 5.4 App Store Connect 素材規格

| 素材 | 規格 |
|------|------|
| App Icon | 1024×1024px，PNG，無圓角，無透明 |
| iPhone 截圖（6.9 吋） | 1320×2868px，至少 3 張 |
| iPhone 截圖（6.5 吋） | 1242×2688px（備選） |
| 副標題 | 最多 30 字元 |
| 關鍵字 | 最多 100 字元，逗號分隔 |
| 促銷文字 | 最多 170 字元 |
| App 描述 | 詳細功能說明 |
| 隱私權政策 URL | 公開可存取 |
| 支援 URL | 可聯絡開發者 |
| App Preview 影片 | 15-30 秒（詳見 `06-SOP-App-Preview.md`） |

### 5.5 隱私營養標籤填寫指南

勾選後 Apple 會對每項問 3 個問題：

| 問題 | 常見回答 |
|------|---------|
| 資料用途？ | **App 功能** |
| 是否與使用者身份關聯？ | 若透過 Auth UID 關聯 → **是** |
| 是否用於追蹤？ | 若未整合追蹤 SDK → **否** |

**各類 App 常見勾選**：
- **無帳號 App**（如 DuoRatio）：選「不收集資料」
- **有帳號 App**（如 Home Bar）：電子郵件地址、使用者識別碼
- **有帳號 + 照片上傳**（如 WedSnap）：電子郵件地址、使用者識別碼、照片或影片

> **原則**：寧可多勾不要少勾，漏報比多報的後果嚴重。

### 5.6 技術合規

| # | 項目 | 說明 |
|---|------|------|
| 1 | **不使用私有 API** | 僅使用公開框架 API |
| 2 | **版本號遞增** | 每次送審必須遞增 version 或 buildNumber |
| 3 | **`ITSAppUsesNonExemptEncryption`** | 若僅用 HTTPS，設為 `false` 免填出口合規文件 |
| 4 | **UIDeviceFamily** | iPhone-only 必須設為 `[1]`。⚠️ Xcode 預設值為 `[1,2]`（含 iPad），未修改會導致 App Store 要求 iPad 截圖 |
| 5 | **最低 iOS 版本** | 根據使用的框架決定（SwiftUI 需 iOS 13+，String Catalogs 需 iOS 16+） |

**UIDeviceFamily 設定方式**：
- Expo：`app.json` → `"supportsTablet": false` + `infoPlist.UIDeviceFamily: [1]`
- Xcode：Build Settings → `TARGETED_DEVICE_FAMILY = 1`

## 七、品牌設計系統框架

> 從 DuoRatio（莫蘭迪色系）、Home Bar（Midnight Lounge）、WedSnap（Soft Neumorphism）三套設計系統提煉。

### 7.1 建立品牌設計系統的步驟

1. **品牌核心**：名稱、標語、使命、願景、品牌個性（3-5 個特質）
2. **色彩計畫**：主色、輔助色、功能色（成功/警告/錯誤），標明使用比例（70/20/10 法則）
3. **字體排印**：標題字型、內文字型、數字字型，字級層次（H1-Body-Caption）
4. **按鈕與互動元素**：按鈕體系（主要/次要/文字/危險）、圖示風格（SF Symbols / Lucide）、微動畫原則
5. **畫面佈局**：留白哲學、排版守則、核心畫面佈局
6. **語氣與文案風格**：語氣光譜、撰寫原則、各情境文案範例

### 7.2 設計系統的程式化

- 建立 Theme 檔案（Swift: `Theme.swift` / JS: `lib/theme.js`）
- 色彩、間距、圓角、動畫參數全部常數化
- 按鈕樣式封裝為可複用元件

### 7.3 設計風格案例參考

| 風格 | App | 特徵 |
|------|-----|------|
| **Midnight Lounge** | Home Bar | 琥珀金 + 深曜黑 + 毛玻璃，適合暗色調品味型 App |
| **Soft Neumorphism** | WedSnap | 米白底 + 柔和陰影 + 圓角，適合溫馨婚禮/社交場景 |
| **莫蘭迪色系** | DuoRatio | 深石青 + 暖米白 + 柔珊瑚，低飽和度，適合專業工具型 App |
